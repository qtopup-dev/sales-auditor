<objective>
Close the single blocking verification gap for Phase 7: `shiftsRouter` mounts no `requireRole` guard, so every shift endpoint (`POST /clock-in`, `POST /clock-out`, `GET /current`, `GET /history`) is reachable by ANY authenticated session — including admins — in direct violation of CLAUDE.md Critical Architecture Rule 9 ("Backend enforces RBAC... Frontend checks are UI only").

Purpose: Shifts are a moderator-only concept (D-05: admins never clock in). Without a server-side guard, an admin session can hit `POST /api/shifts/clock-in` directly and create a `shifts` row for itself; `GET /api/admin/shifts?date=` then joins `shifts→users` with no role filter (`admin.ts` GET /shifts handler) and surfaces that admin as a spurious oversight tab — breaking the "one tab per moderator" invariant central to ROADMAP Success Criterion 4. The fix is the mirror image of an established pattern already in this codebase: `adminRouter.use(requireRole('admin'))`.

Output: `shiftsRouter` gains router-level `requireRole('moderator')` enforcement; the stale "frontend only" comment is removed; the `app.ts` mount comment is corrected to reflect the new guard.
</objective>

| authenticated session → `/api/shifts/*` | An authenticated client (any role) crosses into shift mutation/read endpoints. Before this fix the only gate was `requireAuth` (authentication, not authorization). |

## What Changes With This Fix

This plan adds a single authorization gate — `shiftsRouter.use(requireRole('moderator'))` — at the router level. No route handler logic, data access, or input handling changes. It narrows who may reach the existing handlers; it introduces no new endpoint, parameter, query, or data path.

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |

| T-07-09-01 | Elevation of Privilege | `shiftsRouter` (`POST /clock-in`, `/clock-out`, `GET /current`, `/history`) | mitigate | **Previously exploitable:** any authenticated non-moderator (admin) session could call these endpoints directly (curl/API client) because only `requireAuth` gated them. An admin could create a `shifts` row for itself via `POST /clock-in`. This fix mounts `requireRole('moderator')` at the router level, returning 403 to every non-moderator before any handler runs — closing the privilege-escalation path. |
| T-07-09-02 | Spoofing / Data integrity of oversight dataset | `GET /api/admin/shifts?date=` (consumer of the `shifts` table) | mitigate (transitively) | **Previously exploitable:** the admin oversight query joins `shifts→users` with no role filter, so an admin-created shift row surfaced as a spurious oversight tab, corrupting the "one tab per moderator" invariant (ROADMAP SC4, D-05/D-15). **No new attack surface introduced.** The fix only removes access for a role that was never intended to have it (D-05) and never uses these endpoints through the UI.
