## Color

Inherited from Phase 4/6 without change. Phase 7 adds one clarified distinction (see below) but no new hex values.

| Role | Tailwind Class | Phase 7 Usage |
|------|---------------|----------------|
| Dominant (60%) | `bg-white` / `bg-gray-50` | ShiftTotalsBanner card background (`bg-white`); active tab background (`bg-white`, merges visually with the sheet panel below it); page background (`bg-gray-50`, inherited from layout) |
| Secondary (30%) | `bg-gray-100` | Tab bar strip background (inactive tabs sit on this); table header rows (inherited) |
| Accent (10%) | `blue-600` | Clock In button; Clock Out confirm button (see clarification below); focus rings on all interactive elements |
| Destructive | `red-600` | Force Clock Out button + its confirm dialog button only (see clarification below) |

**Accent (`blue-600`) reserved for in Phase 7:**
1. "Clock In" button in the sidebar ClockControl (primary action, single click, D-09)
2. "Clock Out" confirm button inside `ClockOutConfirmDialog` (see clarification below)
3. Focus rings on all interactive elements: `focus:ring-2 focus:ring-blue-500 focus:ring-offset-2`
4. Active AdminShiftsPage date input border on focus (inherited focus-ring pattern)

**Destructive (`red-600`) reserved for in Phase 7:**
1. "Force Clock Out" button on `AdminShiftsPage` (visible only when viewing today's date and the tab's shift is still open, D-16)
2. "Force Clock Out" confirm button inside `ForceClockOutConfirmDialog`

**Clarification — Clock Out (moderator's own) is NOT destructive-red, Force Clock Out (admin) IS:**
Both actions are gated behind a confirm dialog per D-09/D-16 ("same UX weight as VoidConfirmDialog" = same modal interaction pattern, not necessarily the same color). The two actions differ in consequence:
- A moderator clocking out of their **own** shift is a normal, voluntary, reversible-by-clocking-in-again workflow action — it destroys no data (D-04: voided rows keep shiftId; sales rows are never affected by clock state). It uses the primary **accent** color (`bg-blue-600`), same as any other confirmed primary action.
- An admin force-closing **someone else's** active shift without their consent is an override of another user's live session state — this is the same class of "acting on another party's data without their live input" as Void, so it uses **destructive red** (`bg-red-600`), consistent with the Void button precedent.

## Component Contract: ClockControl

**File:** `packages/frontend/src/components/shift/ClockControl.tsx`

**Status:** NEW.

### Visual Anatomy

```

```

### Props / Data Source

```typescript
interface CurrentShift {
  id: number;
  clockInAt: string; // ISO 8601 UTC
}
```
Data via `useQuery<CurrentShift | null>({ queryKey: ['current-shift'], queryFn: ... })`. `null` response = not clocked in.

### Time Display (D-08)

**Format:** 12-hour with AM/PM, e.g. "Clocked in at 9:03 AM" — derived from the UTC hour/minute components of `clockInAt` (NOT converted to browser-local time). This keeps the display consistent with CLAUDE.md Rule 7 (UTC everywhere) while satisfying the human-readable 12-hour format specified in CONTEXT.md D-08. All other timestamp displays in the app (table columns) use 24-hour UTC "YYYY-MM-DD HH:mm" — this is the one exception, justified because this is a glanceable status readout in a narrow sidebar, not an audit-grade record.

```typescript
function formatClockTime(iso: string): string {
  const d = new Date(iso);
  let hours = d.getUTCHours();
  const minutes = d.getUTCMinutes().toString().padStart(2, '0');
  const ampm = hours >= 12 ? 'PM' : 'AM';
  hours = hours % 12 || 12;
  return `${hours}:${minutes} ${ampm}`;
}
```

**No live-ticking elapsed timer, no client-side interval (D-08 hard constraint).** The displayed time is static once rendered; it does not update until the query refetches (which only happens on mutation success, per D-13's mutation-only refetch philosophy).

### Interaction (D-09, Pessimistic UI per CLAUDE.md Rule 10)

**Clock In — single click, no confirmation:**
1. Moderator clicks "Clock In".
2. Button immediately disabled, label changes to "Clocking In...".
3. `POST /api/shifts/clock-in` (or equivalent — exact path per planner) fires.
4a. Success: `queryClient.invalidateQueries({ queryKey: ['current-shift'] })` → control re-renders in "clocked in" state.
4b. Error: button re-enabled, label reverts to "Clock In", inline error text appears below the button: "Failed to clock in. Please try again."

**Clock Out — requires confirmation (D-09):**
1. Moderator clicks "Clock Out".
2. `useShiftStore().openClockOutDialog()` — opens `<ClockOutConfirmDialog />` (rendered elsewhere, e.g. SalesPage or layout level — Zustand state is global).
3. Confirm/cancel flow handled entirely inside the dialog (see next section) — the sidebar button itself has no separate pending state since the dialog's own confirm button carries the in-flight label.

## Interaction Contracts

### Clock In — Single Click, No Confirmation (D-09)

1. Moderator clicks "Clock In" in the sidebar.
2. Button disables immediately, label → "Clocking In..." (CLAUDE.md Rule 10, pessimistic).
3. `POST` to the clock-in endpoint fires. Backend enforces one-open-shift-per-moderator server-side (D-01) — a duplicate attempt is rejected/no-ops into returning the existing open shift, never surfaced as a user-facing error in normal operation (the button is disabled the entire time a request is in flight, and disappears once clocked-in state renders, so a double-click race is structurally prevented on the happy path).
4a. Success: invalidate `['current-shift']` → ClockControl re-renders showing "Clocked in at {time}" + "Clock Out" button.
4b. Error: button re-enables, label reverts, inline error text shown.

### Clock Out — Confirm Required (D-09)

1. Moderator clicks "Clock Out".
2. `ClockOutConfirmDialog` opens via Zustand `openClockOutDialog()`.
3. Moderator clicks "Clock Out" (confirm) inside the dialog → button shows "Clocking Out..." + disabled, `onClose` set to `undefined` (blocks Escape/backdrop close during the round-trip, mirrors `VoidConfirmDialog`).
4. `POST` to the clock-out endpoint fires.
5a. Success: invalidate `['current-shift']` and the shift-scoped sales query → dialog closes → SalesPage transitions to State A (not clocked in).
5b. Error: dialog stays open, inline error text shown, button re-enabled.
6. "Stay Clocked In" (cancel) — closes dialog with no mutation, only enabled when not mid-request.

### Force Clock Out — Confirm Required, Admin Only (D-16)

1. Admin viewing today's date on `AdminShiftsPage`, on a tab whose shift is still open, clicks "Force Clock Out".
2. `ForceClockOutConfirmDialog` opens with `targetUsername` set from the tab's moderator.
3. Admin clicks "Force Clock Out" (confirm) → button shows "Forcing Clock Out..." + disabled.
4. `POST /api/admin/shifts/:id/force-clock-out` fires — backend sets `clockOutAt = now()` server-side, `requireRole('admin')` enforced (CLAUDE.md Rule 9).
5a. Success: invalidate the admin shifts query for `selectedDate` → dialog closes → the tab's Force Clock Out button disappears (shift now shows a clock-out time) → ShiftTotalsBanner and table remain visible, unaffected (D-04: sales rows untouched by shift closure).
5b. Error: dialog stays open, inline error shown, button re-enabled.

### Shift-Gated Add Row (D-03)

- `hasActiveShift` derived from `['current-shift']` query: `!!currentShift`.
- Add Row button `disabled={!hasActiveShift || isAddRowOpen}`.
- When disabled due to no active shift, native `title` tooltip explains why (no custom tooltip component).
- Inline editing of existing rows is NEVER gated by clock state (D-03) — only the Add Row button and the visible row set (D-11) are shift-aware.

### Per-Shift Sales Sheet Reset (D-11, D-12)

- While clocked in: `SalesPage` queries only rows where `shiftId === currentShiftId`. This is a genuine query-scope change, not a client-side filter over the full history (D-11: "a true reset, not an add-on filter").
- While not clocked in: no sales query runs at all; State A (prompt) renders directly.
- Transition on Clock In success: query key changes from "no query" to the shift-scoped query, naturally triggering a fresh fetch.
- Transition on Clock Out success: shift-scoped query is invalidated/removed, SalesPage reverts to State A.

### Live Shift Totals (D-13)

- `ShiftTotalsBanner` on `SalesPage` derives `count`/`revenue` from the same shift-scoped sales query already used for `SalesTable` (or a dedicated lightweight totals endpoint — planner's discretion) — no separate polling. Updates on Add Row mutation success via the existing `['sales', ...]` invalidation the mutation already performs.
- `ShiftTotalsBanner` on `AdminShiftsPage` derives its values from the per-tab query, which itself polls every 45s while viewing today (D-17) and is static for past dates.

### AdminShiftsPage Date Change

1. Admin changes the date input.
2. `selectedDate` state updates → tab-list query re-fetches for the new date (new query key incorporating the date, e.g. `['admin-shifts', selectedDate]`).
3. `activeShiftId`/selected tab resets to the first tab in the new list (or `null` if no tabs) — the previously selected moderator's tab does not persist across a date change if that moderator has no shift on the new date.
4. `isToday = selectedDate === todayString` recalculates, toggling `refetchInterval` between `45000` and `false`.

| Context | Escape behavior |
|---------|------------------|
| ClockOutConfirmDialog open | Close dialog (unless clock-out in-flight — locked, mirrors VoidConfirmDialog) |
| ForceClockOutConfirmDialog open | Close dialog (unless force-clock-out in-flight — locked) |
