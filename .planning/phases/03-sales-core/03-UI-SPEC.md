## Layout: SalesPage

SalesPage.tsx replaces its placeholder content with a full-height layout:

```
[Page header row: "Sales Sheet" + "Add Row" button]   ← 32px bottom margin
[Virtual scroll sales table — fills remaining height]
[Audit drawer — right slide-in, overlays table]
[Void confirmation dialog — modal overlay]
```

**Page header row:**
- Left: "Sales Sheet" heading (20px/600, text-gray-900)
- Right: "Add Row" button (accent blue, 40px height, 14px/400)
- Row: `flex items-center justify-between mb-8`

**Primary visual anchor:** the "Add Row" button (accent blue, top-right of page header) is the single focal entry point on the page. The sales table is the dominant content area and fills the remaining viewport height below the header.

**Table container:**
- Must be a fixed-height container (not content-height) so the virtualizer can measure it.
- `overflow-hidden` on the outer container; the virtualizer manages internal scroll.
- The main content area is `overflow-auto p-8` (from AuthenticatedLayout); the table container must fill available height: use `flex-1 min-h-0` on the outer wrapper inside `<main>`.

**Mobile (SALES-18):**
- Minimum viable: horizontal scroll on the table container (`overflow-x-auto`) so all columns remain accessible on small screens.
- Column widths remain fixed (see table columns section below); no responsive column hiding in Phase 3.
- The "Add Row" button remains visible and accessible at mobile widths.

### Inline Cell Editing (Existing Rows, D-04, SALES-05, SALES-06)

**Click to edit:**
- Clicking an editable cell focuses it and renders an input in-place (replaces display text).
- Cell transitions: display `<span>` → `<input>` (text fields) or `<textarea>` (Notes field).
- Only one cell active at a time across the entire table (D-05).
- While any cell is actively editing, clicking a different editable cell: current cell blurs (fires PATCH), then new cell becomes active.

**Active cell visual state:**
- Border: `border border-blue-500` (2px, blue-500)
- Background: `bg-white`
- No shadow — border change alone is the signal.
- Input text: 14px/400/text-gray-900.
- Focus ring (on the input element): `focus:outline-none focus:ring-1 focus:ring-blue-500`

**Idle editable cell (hoverable but not editing):**
- `cursor-pointer`
- On hover: `bg-blue-50` background tint — subtle signal that the cell is clickable.
- Border: none (cells have no individual border in idle state; table row border handles separation).

**Idle non-editable cell (Price, Date Edited, or moderator on another's row):**
- `cursor-default`
- No hover state change.
- Text: `text-gray-400`

**Blur / save round-trip (SALES-06, pessimistic UI, D-04):**
1. User blurs the input (clicks elsewhere, tabs out, or presses Enter).
2. Immediately: input becomes disabled (`disabled` attribute), cell background → `bg-gray-100 opacity-60`, display a 16px spinner inline beside the content.
3. PATCH fires to `/api/sales/:id` with `{ field, value }`.
4a. Success: React Query invalidates `['sales']`, cell returns to display mode with updated value.
4b. Error: cell returns to display mode with the previous value, error toast or inline message shown (see Copywriting Contract).

**Save indicator:** 16px inline spinner (CSS animation, `animate-spin`), placed after the cell content, `text-gray-400`. No label needed — disabled state + spinner is sufficient signal.

**Notes cell — textarea specifics:**
- `<textarea>` with `rows={1}` initial and `resize-none` Tailwind class.
- Auto-grows with content: use `onInput` to update height (`element.style.height = element.scrollHeight + 'px'`).
- After blur + save: `measureElement` is called on the row node so the virtualizer recalculates row height (D-01).

**react-select in cell edit mode (Product and MOP columns):**
- Use `react-select` v5 `menuPortalTarget={document.body}` to prevent menu clipping inside the virtual scroll container.
- Menu `z-index`: 9999 (above table, below drawer overlay).
- react-select custom styles aligned to Phase 2 input anatomy: 40px control height, `border-gray-300` → `border-blue-500` on focus, `text-sm`.
- On selection change: fires PATCH immediately (no blur needed for select — change IS the commit action). After PATCH fires, cell returns to display mode.

- "Void" button: `text-red-600 hover:text-red-800 text-sm`, 44px min-height touch target

- Separator: `|` in `text-gray-300`

## Interaction Contracts

### Pessimistic UI — Inline Cell Edit (CLAUDE.md Rule 10, D-04)

1. User blurs editable cell (or selects from react-select combo).
2. Immediately: cell input is `disabled`, cell background → `bg-gray-100 opacity-60`, spinner appears.
3. PATCH fires.
4a. Success: React Query invalidates `['sales']`, cell returns to display mode with new value from server response.
4b. Error: cell returns to display mode with the PREVIOUS value (not the draft), inline error "Save failed. Please try again." shown below the cell for 4 seconds then auto-dismisses.

No optimistic updates. Draft value is in Zustand `salesEditStore` (`draftValue`), not in React Query cache.

### Pessimistic UI — Add Row Form (CLAUDE.md Rule 10, D-02)

1. User clicks "Save Row" in the Add Row form row.
2. Immediately: all inputs in the Add Row row disabled, "Save Row" → "Saving...", "Discard" disabled.
3. POST fires.
4a. Success: React Query invalidates `['sales']`, Add Row form row disappears, virtualizer scrolls to top (new row is first item), new row appears in normal display mode.
4b. Error: all inputs re-enabled, "Save Row" re-enabled, error message displayed below the form row: "Failed to save. Please try again."

### Pessimistic UI — Void Action (CLAUDE.md Rule 10, D-10)

1. Admin clicks "Void" button.
2. Confirmation dialog opens (instant — no in-flight state yet).
3. Admin clicks "Void Row" in dialog.
4. Immediately: "Void Row" button → "Voiding...", disabled; "Keep Row" button disabled.
5. POST fires.
6a. Success: React Query invalidates `['sales']`, dialog closes, row reappears with voided styling.
6b. Error: dialog re-enables both buttons, error shown inside dialog body: "Failed to void row. Please try again."

### One Active Cell at a Time (D-04, D-05)

- Zustand `salesEditStore` tracks `{ activeCellSaleId, activeCellField, draftValue, isPending }`.
- When a cell is clicked: if another cell is already active, blur + PATCH the current cell first, THEN activate the new cell. The new cell does not become editable until the prior PATCH completes.
- This prevents competing PATCH requests and race conditions.
- While `isPending: true` (round-trip in progress): clicking any cell has no effect (cells show `cursor-wait`).

### Add Row Blocks Cell Editing (D-03)

- While `isAddRowOpen: true` in `salesEditStore`: all editable cells on existing rows have no `onClick` handler (or handler is a no-op).
- Visual: existing cells show `cursor-default` (not `cursor-pointer`).
- "Add Row" button: disabled while form is open (`text-gray-400 cursor-not-allowed`).

### Virtual Scroll Behavior (D-01, SALES-02, SALES-03)

- `useVirtualizer` from `@tanstack/react-virtual` v3 with `measureElement` enabled.
- Each row's root DOM node receives `ref={virtualizer.measureElement}` via `data-index` attribute.
- Notes textarea content changes call `measureElement` on the containing row after blur so the virtualizer recalculates height.
- On Add Row save: `scrollToIndex(0)` to bring the new row into view (first position = newest).
- No `estimateSize` accuracy needed beyond a reasonable default (e.g., 56px) — `measureElement` corrects actual heights.

### Escape Key Handling (global within SalesPage)

| Context | Escape behavior |
|---------|----------------|
| Add Row form open | Dismiss Add Row form (same as Discard) |
| Cell actively editing | Discard draft, return cell to display mode (no PATCH) — clear Zustand draft |
| Audit drawer open | Close drawer |
| Void confirmation dialog open (idle) | Close dialog |
| Void in-flight | No-op (dialog locked during round-trip) |

### Audit Drawer — Data Fetch

- Triggered by "Audit" button click.
- `useQuery(['sales', saleId, 'audit'])` fires `GET /api/sales/:id/audit` when drawer opens.
- Loading: "Loading..." text in drawer body.
- Data: render entries newest-first.
- Drawer stays open if user opens it on one row, then clicks "Audit" on another row — it should re-fetch for the new row (key changes, React Query re-fetches automatically).

### react-select Combo Behavior (Product and MOP, SALES-08, SALES-10, PROD-05, PAY-05)

**In Add Row form:**
- `loadOptions` async: `GET /api/products?active=true` or `GET /api/mops?active=true`.
- Minimum characters to trigger search: 0 (load all active on open, then filter client-side for a small catalog).
- `menuPortalTarget={document.body}` to avoid overflow clipping.
- On product selection: immediately populate Price display from `product.price` (string, displayed as-is).

**In cell edit mode (existing row):**
- Same async load behavior.
- `defaultValue` set to current `productNameSnapshot` / `mopNameSnapshot` so user sees the existing value on open.
- On selection: triggers PATCH immediately (change = commit for selects). Cell returns to display mode after PATCH resolves.
- Inactive products/MOPs that are referenced by the existing row: NOT shown in the combo options during edit. Display-only value uses the snapshot string. **Note:** if a moderator edits the Product field on a row that has an inactive product, they cannot re-select that product from the combo — they must select an active one. This is by design (PROD-05, PAY-05).
