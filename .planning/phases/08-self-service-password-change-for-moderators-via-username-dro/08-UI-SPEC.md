## Component Notes — New Surfaces This Phase Introduces

### Username Dropdown Menu (replaces `AuthenticatedLayout.tsx` lines 93-103)

No prior dropdown-menu component exists in the codebase — this is a new pattern. Prescriptive spec:

- **Trigger**: full-width `<button>` inside the existing `px-4 py-4 border-t border-gray-200 dark:border-gray-800` footer container. Shows `{username}` (truncated, `text-sm text-gray-700 dark:text-gray-300`) left-aligned + a small chevron icon (up-pointing when closed, since menu opens upward; rotates or flips when open) right-aligned. `min-h-[44px]`, `hover:bg-gray-200 dark:hover:bg-gray-800`, `rounded`, `focus:outline-none focus:ring-2 focus:ring-blue-500`.
- **Panel**: opens **upward** (username sits at the very bottom of the sidebar — an downward-opening panel would be clipped by the viewport). Positioned `absolute bottom-full left-0 right-0 mb-1` relative to the trigger. `bg-white dark:bg-gray-800`, `border border-gray-200 dark:border-gray-700`, `rounded-md`, `shadow-lg`, `py-1`, `z-50`.
- **Menu items**: each a full-width `<button>`, `text-sm font-normal text-gray-700 dark:text-gray-300`, `px-4 py-2`, `min-h-[44px]`, `hover:bg-gray-100 dark:hover:bg-gray-700`, left-aligned text, no icons (matches existing nav-link/toggle-button convention of icon-only-when-meaningful — plain text is sufficient for 2 menu items).
- **Divider**: a `border-t border-gray-200 dark:border-gray-700` between "Change Password" and "Log Out" to visually separate the two actions (Log Out ends the session; Change Password does not).
- **Close behavior**: click-outside and Escape both close the panel (standard dropdown UX — reuse the `useEffect` + `document.addEventListener('keydown', ...)` pattern already established in `Modal.tsx` and `AuthenticatedLayout.tsx`'s mobile-drawer Escape handler).
- **Animation**: none required — instant show/hide, consistent with `Modal.tsx` (no existing component in this codebase uses transition/animation classes).

### Change Password Modal

Reuses `Modal.tsx` wrapper exactly as `UserModal.tsx` and `ProductModal.tsx` do. No new modal primitive needed.

- Two `type="password"` inputs, styled identically to `UserModal.tsx`'s text input (`w-full border rounded-md px-3 py-2 text-sm ... bg-white dark:bg-gray-800`, red border + ring on error).
- Field-level errors: `text-xs text-red-600 dark:text-red-400 mt-1` directly under each field (matches `UserModal.tsx` pattern).
- Server/generic error: `text-sm text-red-600 dark:text-red-400 mt-1` below the fields (matches `ProductModal.tsx`/`MopModal.tsx`/`ReceiverModal.tsx` pattern).
- Footer (pre-submit): "Cancel" (secondary button style — `border border-gray-300 dark:border-gray-600 bg-white dark:bg-gray-800`) + "Save Password" (primary — `bg-blue-600 hover:bg-blue-700 text-white`). Both `disabled` during `isPending`, matching `UserModal.tsx`.
- Footer (post-success): swaps to a single "Done" button (`bg-blue-600`, matches `ResetPasswordModal.tsx`), and the body swaps from the form to the success message. `onClose` remains blocked only during the in-flight save (`isPending`), and is freely closeable once success state is shown — mirrors `ResetPasswordModal.tsx`'s "always closeable once the result exists" behavior.
