<objective>
Add a self-service password-change endpoint to the existing (unauthenticated-at-mount) `authRouter`. Any logged-in user changes their OWN password. On success the server re-hashes with bcrypt cost 12, updates the current user's row, and invalidates all of that user's OTHER sessions while explicitly preserving the current session (D-06).

Purpose: Backend half of Phase 8 self-service password change. Diverges from the Phase 2 admin-reset flow (`users.ts POST /:id/reset-password`) on exactly one point — the session-invalidation query must exclude the requester's own current session (`req.sessionID`), so they are NOT logged out by their own action.
Output: One new route + one validation array in `packages/backend/src/routes/auth.ts`, plus a new `sessionPool` import.
</objective>

```typescript
// ─── POST /api/auth/change-password ──────────────────────────────────────────
// Phase 8: self-service password change for the CURRENTLY logged-in user (admin or moderator).
// D-01: no role branching — any authenticated user may change their own password.
// D-03: no current-password field (confirmed user decision — internal tool, trusted team).
// D-06: invalidate all of THIS user's OTHER sessions but PRESERVE the current one (req.sessionID).
//       This is the ONE point of divergence from users.ts admin-reset, which kills every session.
// requireAuth is per-route because authRouter is mounted unauthenticated in app.ts (see GET /me).
authRouter.post('/change-password', requireAuth, changePasswordValidation, async (req: Request, res: Response) => {
  const errors = validationResult(req);
  if (!errors.isEmpty()) {
    res.status(400).json({ error: 'VALIDATION_ERROR', details: errors.array() });
    return;
  }

  const { newPassword } = req.body as { newPassword: string };

  // Hash before touching the DB — bcrypt cost 12 per CLAUDE.md (same as invite register + admin reset).
  const passwordHash = await bcrypt.hash(newPassword, 12);

  // Update ONLY the caller's own row — id comes from the session, never from the request body,
  // so a user can never change another account's password (Rule 9 backend RBAC enforcement).
  await prisma.user.update({
    where: { id: req.session.userId!, organizationId: req.session.organizationId! },
    data: { passwordHash },
  });

  // D-06: kill every OTHER session for this user, but keep the current one alive.
  // session_id is the express-session primary key (== req.sessionID); excluding it preserves
  // the requester's login. Cannot reuse the admin-reset query verbatim — that deletes ALL sessions.
  await sessionPool.query(
    `DELETE FROM sessions WHERE JSON_EXTRACT(data, '$.userId') = ? AND session_id != ?`,
    [req.session.userId, req.sessionID],
  );

  // No password is echoed back (unlike admin reset which returns tempPassword) — mirror /logout's shape.
  res.json({ ok: true });
});
```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser → Express (`POST /api/auth/change-password`) | Untrusted request body (`newPassword`) crosses here; identity is taken from the server-side session, never the body |
| Express → sessions table (mysql2 `sessionPool`) | Direct SQL DELETE with the caller's `userId` + `sessionID` as bound params |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-08-01 | Elevation / Broken Access Control | `prisma.user.update` in change-password handler | mitigate | Update `where` uses `req.session.userId!` + `organizationId` from the session ONLY. No `id` is ever read from the request body, so a caller cannot target another account. |
| T-08-02 | Tampering (client-validation bypass) | `changePasswordValidation` | mitigate | Server re-validates `newPassword` with `isLength({ min: 8 })` and returns 400 `VALIDATION_ERROR`; the confirm-match check is client-only and never trusted (Rule 9). |
| T-08-03 | Spoofing (CSRF) | `POST /change-password` state-changing endpoint | mitigate (accept residual) | Existing session cookie is `httpOnly`, `secure` (prod), `sameSite: 'lax'` (app.ts 89-92) — a cross-site forged POST does not carry the cookie. No CSRF-token middleware exists project-wide; residual risk matches every other mutation endpoint and is accepted for this internal tool. |
| T-08-04 | Tampering / Session hijack during invalidation | `sessionPool.query DELETE` | mitigate | Exclusion clause `AND session_id != ?` bound to `req.sessionID` guarantees the current session is preserved and every OTHER session for the user is destroyed (D-06). Params are bound (no SQL injection). |
| T-08-05 | Information Disclosure / Enumeration | change-password response | mitigate | Response is `{ ok: true }` only — no password, no user data echoed. Route requires an authenticated session, so there is no unauthenticated enumeration surface. |
| T-08-06 | Elevation via unattended already-logged-in session | absence of current-password field | accept | Explicit, confirmed user decision (D-03) — internal tool, small trusted team. Do NOT add a current-password field without re-checking with the user (08-CONTEXT.md §Specifics). |

No high-severity threat is left unmitigated; T-08-06 is an accepted decision, not an open finding.
