<objective>
Close the single blocking verification gap for Phase 9 (PHASE9-SC5, CR-01 from 09-REVIEW.md): the race-safe last-admin guard in `DELETE /api/users/:id` counts a deactivated (`isActive: false`, `deletedAt: null`) admin as "another admin," so deleting the sole ACTIVE admin can leave an organization with zero admins able to authenticate — undermining the entire purpose of the safeguard (CONTEXT.md D-09: "if the count would drop to 0, reject").

Fix: add `AND isActive = true` to the `FOR UPDATE` query so the locked set contains only login-capable admins, and gate the guard on `target.isActive` so deleting a deactivated admin — which can never reduce the usable-admin count — is not spuriously blocked.

Purpose: Make the last-admin block enforce its stated invariant — "at least one admin who can actually log in must remain" — instead of merely "at least one admin row must remain."
Output: One modified `if (target.role === 'admin')` block and one modified existence-check `select` in `packages/backend/src/routes/users.ts`. No other route, file, or phase-9 behavior touched.
</objective>

The `DELETE /api/users/:id` route was added in Plan 03. The relevant transaction block CURRENTLY reads:

```typescript
    await prisma.$transaction(async (tx) => {
      // Existence check bypasses isActive (undefined = include inactive-but-not-deleted rows),
      // but KEEPS deletedAt: null enforced — an already-deleted user is treated as not-found.
      const target = await tx.user.findFirst({
        where: { id: targetId, organizationId, isActive: undefined, deletedAt: null },
        select: { id: true, role: true },
      });
      if (!target) {
        throw Object.assign(new Error('User not found'), {
          statusCode: 404, code: 'USER_NOT_FOUND',
        });
      }

      // D-09: race-safe last-admin guard. FOR UPDATE locks ALL of the organization's admin rows
      // ... (comment continues) ...
      if (target.role === 'admin') {
        const admins = await tx.$queryRaw<{ id: number }[]>`
          SELECT id FROM users
          WHERE organizationId = ${organizationId}
            AND role = 'admin'
            AND deletedAt IS NULL
          FOR UPDATE
        `;
        const otherAdmins = admins.filter((a) => a.id !== targetId);
        if (otherAdmins.length === 0) {
          throw Object.assign(new Error('Cannot delete the last remaining admin'), {
            statusCode: 400, code: 'LAST_ADMIN',
          });
        }
      }

      await tx.user.update({
        where: { id: targetId, organizationId },
        data: { deletedAt: new Date() },
      });
    });
```

Key facts the executor needs (all already verified in the codebase — do NOT re-derive):
- The raw SQL uses camelCase column names directly (`organizationId`, `deletedAt`, `role`) — there is NO snake_case @map in this schema. `isActive` is therefore the correct literal column name for the raw query. The existing query type-checks and runs against the live DB with these exact names.
- `isActive` is a Prisma `Boolean` = MySQL `TINYINT(1)`. `AND isActive = true` is valid MySQL (`true` = 1) and is the exact fix form recommended in 09-REVIEW.md CR-01.
- `login` (auth.ts) requires `isActive: true, deletedAt: null` — a deactivated admin genuinely cannot authenticate, which is precisely why a deactivated admin must NOT count toward "a usable admin remains."
- Everything above the transaction (self-delete check via `req.session.userId`) and below it (session-kill `DELETE FROM sessions ...`) is CORRECT and must remain byte-for-byte unchanged. Only the two edits below are in scope.

**Design decision — target's own isActive (the second "missing" item from 09-VERIFICATION.md / CR-01 fix note):** The guard runs ONLY when the target admin is itself active (`target.role === 'admin' && target.isActive`). Rationale: deleting a deactivated (login-incapable) admin can never reduce the count of admins who can log in, so it must never be blocked by a "keep at least one usable admin" invariant. When the target IS active, it is included in the `AND isActive = true FOR UPDATE` locked set, and `admins.filter(a => a.id !== targetId)` yields the count of OTHER usable admins — reject if zero. This preserves race-safety for the only case that matters (deleting active admins): two concurrent deletes of two different active admins lock the identical active-admin set and contend.

| T-09-10 | Denial of Service (last-admin race condition) | Concurrent `DELETE /:id` on two different ACTIVE admins | mitigate | Race-safety is preserved: the `FOR UPDATE` set now covers active admins (target included, not SQL-excluded).
