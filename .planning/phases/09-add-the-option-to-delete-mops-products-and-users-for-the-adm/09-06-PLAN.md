<objective>
Close a gap flagged by gsd-plan-checker during review of Plans 09-01..09-05: `packages/backend/src/routes/sales.ts` performs four product/MOP existence-check queries (`tx.product.findFirst` at the sale-creation path and the `productId`-edit path; `tx.mop.findFirst` at the sale-creation path and the `mopId`-edit path) that filter on `isActive: true` but do NOT filter on `deletedAt: null`. Since `deletedAt` (added in Plan 09-01) and `isActive` are explicitly independent signals per CONTEXT.md D-01, a still-`isActive: true` but soft-deleted product/MOP — correctly hidden from combo boxes by Plan 09-01's `$extends` filter and by Plan 09-04's frontend — could still be successfully referenced to create or edit a sale by any authenticated user who has or guesses its numeric ID.

Purpose: Extend CLAUDE.md Rule 8 (soft-delete filter enforcement) to cover this write-path existence check, not just list/combo-box queries — this is the same class of enforcement Plan 09-01 already added to `findMany`, applied here to four explicit `tx.findFirst` calls that Plan 09-01 could not reach (transactions bypass `$extends`, per the file's own inline comments).
Output: Four `where` clauses in `sales.ts` each gain a `deletedAt: null` condition alongside their existing `isActive: true` condition. No other behavior in the file changes.
</objective>

Route naming/behavior is otherwise unaffected: the existing generic `404 { error: 'NOT_FOUND' }` response and `throw Object.assign(new Error(...), { statusCode, code })` pattern stay exactly as-is for all four blocks — only the `where` clause gains one new key.
