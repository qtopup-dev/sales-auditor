## Component & Interaction Contract

*(Non-template section — included because CONTEXT.md's "Claude's Discretion" list requires concrete UI decisions the planner/executor need before writing code.)*

### New components (3 — one per entity, no shared generic `ConfirmDialog` introduced)

The codebase has no generic reusable confirm dialog today (CONTEXT.md explicit note) — each existing confirmation is hand-built per feature. Follow that convention, not the Zustand-store pattern `VoidConfirmDialog` uses (that pattern exists only because `VoidConfirmDialog` is triggered from inside a virtualized table row). Products/MOPs/Users pages already manage modal target state locally via `useState` (see `modalTarget` in `ProductsPage.tsx`/`MopsPage.tsx`/`UsersPage.tsx`) — the delete dialogs follow that same local-state convention:

| Component | Location | Modeled on |
|-----------|----------|------------|
| `ProductDeleteConfirmDialog.tsx` | `packages/frontend/src/components/catalog/` | `VoidConfirmDialog.tsx` structure (Modal + useMutation + pessimistic pending), but state passed as props (`product: Product \| null`, `onClose`) — not Zustand |
| `MopDeleteConfirmDialog.tsx` | `packages/frontend/src/components/catalog/` | Same as above, `mop: Mop \| null` |
| `UserDeleteConfirmDialog.tsx` | `packages/frontend/src/components/users/` | Same shape, `user: User \| null`; additionally maps `err.response?.data?.code` → the two specific error strings above (`CANNOT_DELETE_SELF`, `LAST_ADMIN`) before falling back to the generic message |

Each dialog:
- Wraps `Modal.tsx` (`open`, `onClose` — `undefined` while pending, blocking escape/backdrop-close per Rule 10, `title`, `footer`)
- Footer: `Cancel` button (existing gray/white bordered style, exact classes below) + `Delete {Entity}` button (existing red style, exact classes below)
- Body: single `<p className="text-sm font-normal text-gray-900 dark:text-gray-100">` with the entity-specific copy above, plus conditional error `<p>` below it
- On success: `queryClient.invalidateQueries({ queryKey: ['products'] })` (or `['mops']` / `['users']`) — the row disappears because the backend now excludes `deletedAt IS NOT NULL` rows (D-02); no client-side optimistic removal (Rule 10)

Exact Tailwind classes (copy verbatim from `VoidConfirmDialog.tsx`, only label text changes):

```
Cancel button:
"px-4 py-2 border border-gray-300 dark:border-gray-600 bg-white dark:bg-gray-800 text-gray-700 dark:text-gray-300 rounded-md text-sm font-normal hover:bg-gray-50 dark:hover:bg-gray-700 disabled:opacity-50 disabled:cursor-not-allowed"

Confirm (destructive) button:
"px-4 py-2 bg-red-600 text-white rounded-md text-sm font-normal hover:bg-red-700 disabled:opacity-50 disabled:cursor-not-allowed"
```

### Actions column changes

**Products (`ProductsPage.tsx`) / MOPs (`MopsPage.tsx`)** — Actions column becomes `Edit | Activate/Deactivate | Delete`:

```
<span className="text-gray-300 dark:text-gray-600 mx-1">|</span>
<button
  type="button"
  disabled={isThisTogglePending}
  onClick={() => setDeleteTarget(product)}
  className="text-red-600 dark:text-red-400 hover:text-red-800 dark:hover:text-red-300 text-sm disabled:text-gray-400 disabled:cursor-not-allowed"
>
  Delete
</button>
```

**Users (`UsersPage.tsx`)** — Actions column becomes `Edit | Enable/Disable Editing | Reset Password | Delete`, same red classes, added after the existing Reset Password button with the standard `|` separator.

New local state per page: `deleteTarget: Product | null` (or `Mop | null` / `User | null`) — mirrors the existing `modalTarget` pattern exactly, kept separate from `modalTarget` so Edit and Delete can never collide.

**Own-row Delete on Users page (UX recommendation, not explicitly locked in CONTEXT.md — backend D-08 is the actual enforcement):** disable the Delete link (same `disabled:text-gray-400 disabled:cursor-not-allowed` treatment already used elsewhere) when `user.id === useAuthStore.getState().user?.id`, to avoid a guaranteed round-trip to a 400 the admin can never succeed at. This is additive UI polish; the backend block (D-08) is authoritative regardless of whether the frontend disables the button.

### Row-level pending state

Clicking `Delete` does not itself trigger an async call — it only opens the confirm dialog (`setDeleteTarget(entity)`), synchronous, no pending state needed at click time. The pending/disabled state lives entirely inside the confirm dialog's own buttons during the mutation round-trip, exactly like `VoidConfirmDialog`. The row's other action buttons (`Edit`, `Activate/Deactivate`, `Reset Password`) should remain disabled while `deleteTarget` is set on that same row (reuse the existing `isThisTogglePending` / `isRowPending` disable pattern, extended to also check the delete-in-flight state) to prevent overlapping mutations on one row.

### No restore / no "show deleted" UI (D-07)

No new page, tab, filter, toggle, or button anywhere renders deleted rows. This is a strict "nothing to build" boundary — do not add any UI affordance for viewing or restoring deleted Products/MOPs/Users.
