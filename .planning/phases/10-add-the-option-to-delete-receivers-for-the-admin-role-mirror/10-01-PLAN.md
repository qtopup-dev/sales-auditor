Output: `deletedAt` nullable column + composite `(organizationId, deletedAt)` index on the `receivers` table in the live database; regenerated Prisma client; extended `$extends` `receiver` filter in `prisma.ts`.

Current Receiver model (lines 110-124 today):
```prisma
model Receiver {
  id             Int          @id @default(autoincrement())
  organizationId Int
  organization   Organization @relation(fields: [organizationId], references: [id])
  name           String       @db.VarChar(255)
  accountNumber  String?      @db.VarChar(100)
  isActive       Boolean      @default(true)
  createdAt      DateTime     @default(now())
  updatedAt      DateTime     @updatedAt

  sales Sale[]

  @@index([organizationId, isActive])
  @@map("receivers")
}
```

Current prisma.ts `$extends` softDeleteFilter block (full file, 73 lines). The `receiver` block is the ONLY block this plan changes — it currently injects `isActive: true` but NOT `deletedAt: null`. The `user`/`product`/`mop` blocks ALREADY inject `deletedAt: null` (Phase 9) and must be left untouched:
```typescript
    // Receivers: inject isActive=true as default (catalog endpoint uses this to hide inactive)
    receiver: {
      findMany({ args, query }) {
        args.where = { isActive: true, ...args.where };
        return query(args);
      },
    },
```

CRITICAL nuance (Phase 9 D-02, mirrored here): today's admin `GET /api/receivers` route (packages/backend/src/routes/receivers.ts line ~35) overrides the `isActive: true` default with `isActive: undefined` (a hardcoded literal in server code) so the admin table shows both active and inactive receivers. The `GET /api/catalog/receivers` route (packages/backend/src/routes/catalog.ts line ~31) relies on the `$extends` default with no override. When adding `deletedAt: null` to the `$extends` `receiver` default, do NOT add any `deletedAt: undefined` override anywhere — `deletedAt: null` must remain unconditionally enforced with zero override path, so deleted receivers vanish from BOTH the admin list and the combo box (there is no "show deleted" surface).

Existing manual migration workflow precedent (packages/backend/prisma/migrations/20260721081035_add-deleted-at-soft-delete/migration.sql from Phase 9, and 20260626091954_add-receiver-catalog/migration.sql from Phase 5) — STATE.md locked decision: "Manual migration workflow (db execute + migrate resolve) — sessions table drift blocks `prisma migrate dev`." Do NOT run `prisma migrate dev` or `prisma migrate dev --create-only` — create the migration directory and SQL file by hand, apply with `prisma db execute`, then register with `prisma migrate resolve --applied`.
