## Design System

| Property | Value |

### Axios 401 Interceptor Behavior

Not a visual component — but defines observable UX:
- On 401 response from any `/api` endpoint: clear React Query cache, navigate to `/login` with `location.state = { returnTo: currentPath }`
- On `/login` page: if `location.state.returnTo` exists, redirect to that path after successful login; otherwise use role-based default (Admin → `/dashboard`, Moderator → `/sales`)
- No visible toast or error on the intercept redirect — the login page itself handles the flow

## Interaction Contracts

### Pessimistic UI (per CLAUDE.md Rule 10)

Every form submit and every row action follows pessimistic UI:

1. User triggers action (click Save, click Deactivate/Activate)
2. Immediately: disable all interactive elements in scope (form inputs, modal buttons, action buttons on that row)
3. Show loading indicator (button label changes to in-flight copy)
4. Await server response
5a. Success: close modal (if modal) OR update row in-place via React Query cache invalidation; re-enable elements
5b. Error: re-enable elements, display inline error message, keep modal open

No optimistic cache updates. No skeleton loaders for actions. Loading state is strictly the disabled + label-change pattern.

### Modal open/close

- Open: triggered by "Add" button or row click/Edit button
- Close: "Cancel" button, X button, Escape key, clicking overlay backdrop
- Close is blocked while a save is in-flight (disabled state)
- No animation required for Phase 2 — simple show/hide

### Form validation timing

- Validate on submit (not on blur for Phase 2 — reduces noise in simple forms)
- Display all field errors at once after failed submit
- Clear individual field error as soon as user modifies that field

### Session expiry UX

- 401 interceptor fires → silent redirect to `/login` with returnTo stored in location state
- No toast notification on intercept — the redirect itself is the signal
- After re-login, user lands at their original destination

### Soft-delete visibility rule

- Inactive products and MOPs: visible in admin catalog table WITH gray "Inactive" badge
- Inactive items: do NOT appear in combo box options (Phase 3 concern — documented here as contract for Phase 3 to honor)
- Toggle deactivate/activate: optimistic label change NOT used — wait for server confirmation before updating badge
