# Phase 4 — UI Design Contract: Admin Dashboard + Management

> Visual and interaction contract for Phase 4. Generated by gsd-ui-researcher.
> This document inherits all Phase 3 design system decisions and extends them with
> new interaction patterns (stats banner, Recharts charts, filter bar, admin all-sales table,
> CSV export, user management CRUD with invite/reset modals).
> A developer must be able to implement the full Phase 4 frontend from this document alone.

## Spacing Scale

Inherited from Phase 3 without change:

| Token | Value | Phase 4 Usage |
|-------|-------|---------------|
| xs | 4px | Gap between stat card label and value, gap between action buttons in users table |
| sm | 8px | Padding inside stat card, gap between filter controls in filter bar, table cell vertical padding |
| md | 16px | Table cell horizontal padding, modal body field spacing, chart margin |
| lg | 24px | Section spacing (stats → charts → table), page header bottom margin, modal section padding |
| xl | 32px | Page header bottom margin (matching ProductsPage pattern) |
| 2xl | 48px | Vertical gap between DashboardPage sections (stats banner, charts area, filter bar + table) |
| 3xl | 64px | Not used in Phase 4 |

**Phase 4 exceptions:**
- Stat card minimum width: 200px (functional constraint — displays "Total Revenue: ₱10,500.00").
- Chart container height: 256px (`h-64` via Tailwind) — explicit height required for ResponsiveContainer to render (see Pitfall 1 in RESEARCH.md). Chart SVG fills 100% width × 256px height.
- Filter bar: horizontal flex row, no forced height. Controls are 40px height (h-10 — matching button height across the app).
- AuditDrawer: 400px wide (inherited from Phase 3 — unchanged).
- Touch targets (all action buttons in users table): minimum 44px effective target.

## Typography

Inherited from Phase 3 without change. Two weights only — 400 and 600.

| Role | Size | Weight | Line Height | Tailwind | Phase 4 Usage |
|------|------|--------|-------------|----------|---------------|
| Body | 14px | 400 | 1.5 | `text-sm font-normal text-gray-900` | Table cell values, modal form values, filter dropdown labels, chart tick labels, stat card values |
| Label | 14px | 400 | 1.4 | `text-sm font-normal text-gray-500` | Column headers, filter input labels, modal field labels, chart axis ticks |
| Heading | 20px | 600 | 1.3 | `text-xl font-semibold text-gray-900` | "Dashboard" page title, "Users" page title, modal titles |
| Display | 24px | 600 | 1.2 | `text-2xl font-semibold text-gray-900` | Stat card values (Total Sales count, Total Revenue amount) |

**No weight 500 (`font-medium`) anywhere in Phase 4.** All elements use 400 or 600.

**Stat card value exception:** stat card values use Display (24px/600/gray-900) — the only place Display is used in Phase 4. This distinguishes them as KPI numbers within the page hierarchy.

**Chart tick labels:** 12px/400/gray-500 (`text-xs font-normal text-gray-500`) — applied via Recharts `tick={{ fontSize: 12 }}` prop. Smaller than body because chart space is constrained.

## Layout: DashboardPage

DashboardPage.tsx replaces its placeholder with a full admin hub. Single page covering ADMIN-01 through ADMIN-12 (D-01).

```

[Stats banner: 2 stat cards side by side]                                   ← mb-12 (48px below)

[Admin all-sales table]
[AuditDrawer — right slide-in, overlays all content]
[VoidConfirmDialog — modal overlay]
```

**Page header row:** `flex items-center justify-between mb-6`
- Left: "Dashboard" heading — `text-xl font-semibold text-gray-900`
- Right: "Export CSV" button — `px-4 py-2 h-10 bg-blue-600 text-white rounded-md text-sm font-normal hover:bg-blue-700 focus:outline-none focus:ring-2 focus:ring-blue-500 focus:ring-offset-2`

**Primary visual anchor:** The stats banner at the top provides the admin's immediate KPI read (total sales + revenue). The "Export CSV" button is the page's primary CTA and sits top-right of the header per established page pattern. Charts follow below stats. The filterable table below charts is the primary interaction area.

**Stats banner layout:** `grid grid-cols-2 gap-6` (two columns, 24px gap between cards).

**Charts area layout:** `grid grid-cols-3 gap-6` (three columns on desktop, each chart in its own card). On viewports narrower than 900px, fall back to `grid-cols-1` so charts stack vertically.

**Filter bar layout:** Horizontal flex row — `flex items-end gap-4 flex-wrap`.

**Admin sales table:** Standard border/overflow table (same pattern as ProductsPage). No virtual scroll — admin table is read-only and not expected to exceed page-scroll threshold in v1 (D-04: client-side in-memory).

**Note on summary stats vs filter:** Summary stats and charts reflect the FULL dataset (all org data, not affected by client-side filters). The filter bar affects the admin table and CSV export only. This distinction is not surfaced in UI copy — it is an architectural decision (D-09) that is invisible to the user.

## Component Contracts

### StatCard

Two stat cards rendered side by side in the stats banner. Each card is a white bordered container showing a label + large number.

**Visual anatomy:**
```
[bg-white border border-gray-200 rounded-md p-6]
  [Label — 14px/400/gray-500, mb-1]
  [Value — 24px/600/gray-900]
```

Tailwind: `bg-white border border-gray-200 rounded-md p-6`

| Stat | Label | Value format | Source |
|------|-------|--------------|--------|
| Total Sales | "Total Sales" | Integer, e.g. "42" | `summary.totalCount` |
| Total Revenue | "Total Revenue" | String with currency prefix, e.g. "₱10,500.00" | `summary.totalRevenue` (string from server) |

**Revenue display note:** `summary.totalRevenue` arrives as a string (e.g., `"10500.00"`) from the server. Display as `"₱" + summary.totalRevenue`. Do NOT parse it as a float — Rule 6 (CLAUDE.md).

**Loading state:** While `summaryLoading` is true, render skeleton placeholders — two `animate-pulse bg-gray-200 h-8 rounded` divs in each card value position. No spinner — pulse skeleton matches the card value area shape.

**Error state:** If summary fetch fails, render "—" in place of both values. No error text in the card — the filter bar area can surface a general error message.

### SalesCharts

Three Recharts charts in a `grid grid-cols-3 gap-6` layout. Each chart is wrapped in a white bordered card.

**Chart card anatomy:**
```
[bg-white border border-gray-200 rounded-md p-4]
  [Chart title — 14px/400/gray-900, mb-3]
  [div h-64 — explicit height container for ResponsiveContainer]
    [ResponsiveContainer width="100%" height="100%"]
      [LineChart / BarChart]
```

Tailwind for chart card: `bg-white border border-gray-200 rounded-md p-4`
Chart title: `text-sm font-normal text-gray-900 mb-3`
Height container: `h-64` (256px) — critical to prevent ResponsiveContainer collapse to 0px.

**Chart 1 — Sales Trend (LineChart):**
- Title: "Sales Over Time"
- Data key: `trendData` from `summary.trendData` — shape `{ date: string; count: number }[]`
- X axis: `dataKey="date"`, `tick={{ fontSize: 12 }}`, no label
- Y axis: `allowDecimals={false}`, `tick={{ fontSize: 12 }}`
- Grid: `CartesianGrid strokeDasharray="3 3" stroke="#e5e7eb"`
- Line: `dataKey="count"`, `stroke="#2563eb"` (blue-600), `strokeWidth={2}`, `dot={false}`
- Tooltip: default Recharts Tooltip (no customization)
- Margin: `{ top: 5, right: 20, left: 0, bottom: 5 }`
- Empty state: if `trendData` is empty, render "No data yet." (14px/400/gray-500) in place of the chart inside the `h-64` container.

**Chart 2 — Product Breakdown (BarChart):**
- Title: "Sales by Product"
- Data key: `productBreakdown` from `summary.productBreakdown` — shape `{ name: string; count: number; revenue: string }[]`
- X axis: `dataKey="name"`, `tick={{ fontSize: 12, angle: -30, textAnchor: 'end' }}`, bottom margin 40px (`margin={{ bottom: 40 }}`)
- Y axis: `allowDecimals={false}`, `tick={{ fontSize: 12 }}`
- Grid: `CartesianGrid strokeDasharray="3 3" stroke="#e5e7eb"`
- Bar: `dataKey="count"`, `fill="#2563eb"`, `radius={[3, 3, 0, 0]}`
- Tooltip: default Recharts Tooltip
- Margin: `{ top: 5, right: 20, left: 0, bottom: 40 }`
- Empty state: "No data yet." in place of chart.

**Chart 3 — MOP Breakdown (BarChart):**
- Title: "Sales by Payment Method"
- Data key: `mopBreakdown` from `summary.mopBreakdown` — shape `{ name: string; count: number }[]`
- Identical configuration to Chart 2 (same bar chart pattern, different data source).
- Empty state: "No data yet." in place of chart.

**Loading state:** While `summaryLoading` is true, render `animate-pulse bg-gray-200 rounded h-64 w-full` in each chart card body. Three pulse blocks — one per chart position.

**Responsive:** On viewports narrower than 900px, the three-column grid collapses to single column via responsive class: `grid grid-cols-1 md:grid-cols-3 gap-6`. `md` breakpoint is 768px in Tailwind v3.

### SalesFilterBar

Always-visible horizontal bar above the admin all-sales table. No "Apply" button — filters are applied live on each control change (D-05).

**Layout:** `flex items-end gap-4 flex-wrap mb-4`

Each filter control is a vertical stack: label above, input below.

```
[Label text: 14px/400/gray-500]
[Input / Select: h-10 (40px)]
```

Label wrapper: `flex flex-col gap-1`
Label: `text-sm font-normal text-gray-500`

**Filter controls:**

| Control | Type | Label | Tailwind / Config | Clears to |
|---------|------|-------|-------------------|-----------|
| Start Date | `<input type="date">` | "From" | `h-10 border border-gray-300 rounded-md px-3 text-sm text-gray-900 focus:outline-none focus:ring-2 focus:ring-blue-500` | `""` (empty string → `null` in state) |
| End Date | `<input type="date">` | "To" | Same as Start Date | `""` → `null` |
| Product | react-select v5 | "Product" | `isClearable`, `placeholder="All products"`, height `h-10`, width `w-48`, react-select custom styles (see below) | `null` |
| MOP | react-select v5 | "MOP" | `isClearable`, `placeholder="All MOPs"`, width `w-40` | `null` |
| Moderator | react-select v5 | "Moderator" | `isClearable`, `placeholder="All moderators"`, width `w-48` | `null` |
| Clear All | `<button>` | — | `text-sm text-gray-500 hover:text-gray-900 self-end h-10 px-2`, label "Clear filters" | Resets all to null |

**react-select custom styles for filter bar** (consistent with existing react-select usage in AddRowForm):
```
control: (base) => ({
  ...base,
  height: '40px',
  minHeight: '40px',
  borderColor: '#d1d5db',   // gray-300
  borderRadius: '6px',
  fontSize: '14px',
  '&:hover': { borderColor: '#9ca3af' },  // gray-400
}),
placeholder: (base) => ({ ...base, color: '#6b7280' }),   // gray-500
```
Focus ring: `focus:ring-2 focus:ring-blue-500` — applied via `menuPortalTarget={document.body}` ensures menu does not clip inside the page container.

**Filter state type** (D-06):
```typescript
interface FilterState {
  startDate: string | null;
  endDate: string | null;
  productId: number | null;
  mopId: number | null;
  createdById: number | null;
}
```
Initial state: all nulls (show all rows).

**Live filtering:** Each control's `onChange` updates the corresponding field in FilterState. The `filteredRows` array is derived via `useMemo` over the full sales array — no re-fetch triggered. `filteredRows` is passed to AdminSalesTable and also to the CSV export function.

### AdminSalesTable

Read-only table built with `@tanstack/react-table` v8. No inline editing — admin edits rows via `/sales`. Per-row actions: Audit + Void.

**Column definitions:**

| Column | Width | Alignment | Notes |
|--------|-------|-----------|-------|
| Product | 160px | Left | `productNameSnapshot` string |
| Price | 100px | Right | `priceSnapshot` string; display as-is (no parseFloat) |
| MOP | 140px | Left | `mopNameSnapshot` string |
| Receiver | 140px | Left | `receiver` string |
| Notes | flex-1 (min 120px) | Left | `notes` string; truncate to 2 lines (`line-clamp-2`); no dynamic height (admin table is read-only) |
| Created By | 120px | Left | `createdByUsername` string |
| Created At | 140px | Left | ISO string formatted as "YYYY-MM-DD HH:mm"; `text-gray-500` |
| Last Edited By | 120px | Left | `lastEditedByUsername ?? "—"` |
| Date Edited | 140px | Left | `updatedAt` ISO string formatted as "YYYY-MM-DD HH:mm"; `text-gray-500`. Show "—" if same as createdAt (no edits). |
| Status | 90px | Center | Render status badge: green "Active" or red "Void" (see Color section) |
| Actions | 120px | Center | Audit + Void buttons |

**Notes truncation:** `overflow-hidden` + CSS `display: -webkit-box; -webkit-line-clamp: 2; -webkit-box-orient: vertical` via Tailwind `line-clamp-2`. Full content visible on hover via native `title` attribute (no custom tooltip needed for internal tool).

**Table anatomy** (same as ProductsPage):
- Outer wrapper: `border border-gray-200 rounded-md overflow-hidden`
- `<table className="w-full">`
- Header: `<tr className="bg-gray-100 border-b border-gray-200">`, `<th className="px-4 py-3 text-sm font-normal text-gray-500 text-left">`
- Body row: `<tr className="bg-white border-b border-gray-200 hover:bg-gray-50">`
- Cell: `<td className="px-4 py-3 text-sm text-gray-900">`
- Voided row override: row classname adds `bg-red-50` (`<tr className="bg-red-50 border-b border-gray-200 hover:bg-red-100">`)

**Price column:** `<span className="block text-right text-sm text-gray-900">{row.priceSnapshot}</span>` — right-aligned, display string as returned from API.

**Date columns:** Format ISO string to "YYYY-MM-DD HH:mm" using a simple `formatDateTime` utility. Render in `text-gray-500` (secondary information). Example: `"2026-06-25T10:30:00.000Z"` → `"2026-06-25 10:30"`.

**Actions column — per row:**
- If row is active:
  - "Void" button: `text-red-600 hover:text-red-800 text-sm`, min-height 44px
  - Separator: `<span className="text-gray-300 mx-1">|</span>`
  - "Audit" button: `text-blue-600 hover:text-blue-800 text-sm`, min-height 44px
- If row is voided:
  - "Void" button is hidden (replaced by the Status badge which already shows "Void")
  - "Audit" button remains: `text-blue-600 hover:text-blue-800 text-sm`

**Void action from admin table:** Clicking "Void" on an active row opens `VoidConfirmDialog` (reuse Phase 3 component exactly). On confirm, calls existing `POST /api/sales/:id/void`. After success, `queryClient.invalidateQueries({ queryKey: ['sales'] })`.

**Audit action from admin table:** Clicking "Audit" calls `useSalesEditStore().openAuditDrawer(row.original.id)`. Then `<AuditDrawer />` is rendered at DashboardPage level — it reads from the same Zustand store used in SalesPage. No refactor of AuditDrawer required (D-04 / Option A in RESEARCH.md Pattern 5).

**Empty state (no rows after filtering):**
```
<div className="border border-gray-200 rounded-md p-8 text-center">
  <p className="text-sm font-semibold text-gray-900 mb-1">No sales match the current filters</p>
  <p className="text-sm text-gray-500">Adjust or clear the filters to see results.</p>
</div>
```

**Empty state (no rows at all — fresh org):**
```
<div className="border border-gray-200 rounded-md p-8 text-center">
  <p className="text-sm font-semibold text-gray-900 mb-1">No sales yet</p>
  <p className="text-sm text-gray-500">Sales entered by moderators will appear here.</p>
</div>
```

**Loading state:** "Loading sales..." (14px/400/gray-500) — same pattern as Phase 3.

**Error state:** "Failed to load sales. Please refresh the page." (14px/400/gray-500).

### UserModal

Modal for editing a user's username. Follows `ProductModal.tsx` pattern exactly. Uses existing `Modal.tsx` wrapper.

**Trigger:** "Edit" button in users table actions column.

**Modal props:**
- `open`: boolean
- `onClose`: `undefined` while save is in-flight (blocks Escape + backdrop close per Modal.tsx API), `() => setModalTarget(null)` otherwise
- `title`: "Edit User"
- `footer`: Discard Changes + Save Username buttons

**Form anatomy (react-hook-form v7):**
```
[Label: "Username"]
[Input: text, pre-filled with user.username]
[Inline error if USERNAME_TAKEN: "Username already taken."]
[Inline error if empty/too short: "Username must be at least 2 characters."]
```

Label: `text-sm font-normal text-gray-500 mb-1`
Input: `w-full border border-gray-300 rounded-md px-3 py-2 text-sm text-gray-900 focus:outline-none focus:ring-2 focus:ring-blue-500 focus:border-blue-500`
Error text: `text-xs text-red-600 mt-1`
Input error state border: `border-red-500`

**Footer buttons:**
- "Discard Changes": `border border-gray-300 bg-white text-gray-700 hover:bg-gray-50 px-4 py-2 h-10 rounded-md text-sm`
- "Save Username": `bg-blue-600 text-white hover:bg-blue-700 px-4 py-2 h-10 rounded-md text-sm`
  - In-flight: "Saving Username..." + disabled + `onClose` → `undefined`

## Interaction Contracts

### Pessimistic UI — canEdit Toggle (CLAUDE.md Rule 10, D-19)

1. Admin clicks "Enable Editing" / "Disable Editing" for a moderator row.
2. Immediately: `pendingCanEditId` set to `user.id`. All three action buttons in that row disabled. Button shows "Enabling..." or "Disabling...".
3. `PATCH /api/users/:id` fires with `{ canEdit: boolean }`.
4a. Success: `queryClient.invalidateQueries({ queryKey: ['users'] })`. `pendingCanEditId` cleared.
4b. Error: `pendingCanEditId` cleared, buttons re-enabled. No visible error on row (silent — use console.error; adding a row-level error message is v2 UX polish).

### Pessimistic UI — Reset Password (CLAUDE.md Rule 10, D-18)

1. Admin clicks "Reset Password" for a user row.
2. Immediately: `pendingResetId` set to `user.id`. All three action buttons in row disabled. Button shows "Resetting...".
3. `POST /api/users/:id/reset-password` fires.
4a. Success: store temp password from response → `pendingResetId` cleared → `ResetPasswordModal` opens.
4b. Error: `pendingResetId` cleared, buttons re-enabled. `alert('Failed to reset password. Please try again.')`.

### Invite Moderator — Button State

1. Admin clicks "Invite Moderator".
2. Button shows "Generating..." + disabled.
3. `POST /api/auth/invite-generate` fires.
4a. Success: store invite URL → InviteModal opens → button returns to "Invite Moderator".
4b. Error: button returns to "Invite Moderator". `alert('Failed to generate invite link. Please try again.')`.

### Clipboard Copy — Copy Link / Copy Password

1. User clicks "Copy Link" or "Copy Password".
2. `navigator.clipboard.writeText(value)` — no try/catch needed (internal tool, modern browser assumed).
3. Button label changes to "Copied!" immediately.
4. After 2 seconds (`setTimeout`): label reverts to original.

### Filter Application — Live (D-05)

- Each filter control change calls the relevant `setFilter` action.
- `filteredRows` is derived via `useMemo([sales, filterState])` — no effect, no re-fetch.
- Applied client-side only. The stats banner and charts are NOT affected by filter state (D-09).

### CSV Export Flow (D-10, D-11, D-12, D-13)

1. Admin clicks "Export CSV".
2. Button shows "Exporting..." + disabled (brief — sync operation).
3. `@json2csv/plainjs` `Parser.parse(filteredRows)` runs synchronously.
4. UTF-8 BOM prepended. `Blob` created. `URL.createObjectURL` + synthetic `<a>` click triggers download.
5. `URL.revokeObjectURL` called after click.
6. Button returns to "Export CSV" + enabled.
7. On error: `alert('CSV export failed. Please try again.')`.

**Filter awareness:** The CSV export downloads exactly what is currently visible in the admin table (filteredRows). If no filters are applied, all rows are exported.

### AuditDrawer Open from Admin Table (RESEARCH.md Pattern 5, Option A)

1. Admin clicks "Audit" on a row in the admin sales table.
2. `useSalesEditStore().openAuditDrawer(row.original.id)` is called.
3. `<AuditDrawer />` rendered at DashboardPage level reads `openAuditSaleId` from Zustand and renders.
4. React Query key `['sales', saleId, 'audit']` triggers fetch for that row's audit entries.
5. Close: Escape key, overlay click, or X button — calls `closeAuditDrawer()` from store.

**Dependency:** `salesEditStore.openAuditDrawer(saleId)` action must exist. Planner must verify this action in `salesEditStore.ts` and add it if missing (RESEARCH.md Pitfall 6).

### Escape Key Handling — Phase 4 Additions

| Context | Escape behavior |
|---------|----------------|
| UserModal open | Close modal (unless save in-flight — modal locked) |
| InviteModal open | Close modal |
| ResetPasswordModal open | Close modal |
| AuditDrawer open (from admin table) | Close drawer |
| Void dialog open | Close dialog (unless void in-flight) |
